# 字幕重複出力の修正記録 (2026-03)

## 症状

同じ話者の発話が、1秒ごとに**全文を含む形で**大量に重複出力される。
例えば1文の発話が30行以上のトランスクリプトエントリに分割され、それぞれが全文を含む。

```
話者A 12:00:55
お疲れさまです。えっと...今回の件についてちょっと 確認。
話者A 12:00:56
お疲れさまです。えっと...今回の件についてちょっと確認したいので。
話者A 12:00:56
お疲れさまです。えっと...今回の件についてちょっと確認したいので なん？
...（同じテキストが1文字ずつ伸びて30行以上続く）
```

理想的な出力は、句点（`。`）などの文の区切りで適度にcommitされつつ、重複がないこと。
1つの発話全体が1エントリにまとまる必要はなく、適切な粒度で分割されていればよい。

## 原因の仮説（3つ）

### 仮説1: `MAX_SEGMENT_LENGTH=100` による即時finalize（確度: 高）

100文字を超えるテキストに対して`finalizeDelay = 0ms`を設定していた。
Google Meetの音声認識は約100ms〜1秒ごとにテキストを更新するため、0ms delayでは更新のたびにfinalizeが実行される。
日本語の発話は100文字を簡単に超えるため、ほぼ全ての長い発話でこの問題が発生する。

**修正**: `MAX_SEGMENT_LENGTH`の即時finalize機能を削除。常にデバウンス（通常2500ms / 句読点500ms）を使用。

### 仮説2: 音声認識のテキスト修正で`computeTextDelta`の`startsWith`が失敗（確度: 高）

Google Meetの音声認識は認識中にテキストを修正する。具体的には：

- スペースの追加/削除: `"についてちょっと 確認"` → `"についてちょっと確認"`
- 語尾の変化: `"確認。"` → `"確認したいので。"`
- キャプションウィンドウの拡大: 前方にテキストが追加される

これらの修正が入ると`newText.startsWith(lastText)`が`false`になり、deltaが計算できなかった。

**修正**: `computeTextDelta`に2つのフォールバックを追加。

1. lastText（末尾句読点除去）をnewText内で部分文字列検索
2. スペースを正規化した上での部分文字列検索

### 仮説3: deltaが取れない場合のフルテキスト再出力（確度: 高）

`finalizeSegment`で`computeTextDelta`が`null`を返すと、`?? text`でフルテキストにフォールバックしていた。
つまり、仮説2でdelta計算が失敗するたびに、全文がトランスクリプトに再追加されていた。

**修正**: deltaが取れない場合、appendをスキップして`finalizedSegments`の更新のみ行う。
次回のdelta計算は更新された最新テキストをベースに行われるため、テキストが伸びた時点で正しくdeltaが取れる。

## 残っている懸念・未検証事項

### deltaスキップによるテキスト欠損の可能性

deltaが計算できずappendがスキップされた場合、その間に追加されたテキストが欠損する可能性がある。
ただし`finalizedSegments`は常に最新テキストで更新されるため、次回のdeltaはスキップされたテキストをベースに計算される。

- **最悪ケース**: テキストが毎回修正され続けると、deltaが常に`null`でappendが一度も起きない
- **期待**: 話者が一時停止した時点（2.5秒更新なし）で音声認識が安定し、正しいdeltaが取れる

### 句読点delay(500ms)がまだ早すぎる可能性

音声認識が一時的に`。`を付けてすぐ取り消すケースでは、500msでもfinalizeが早すぎる可能性がある。
もし再び重複が見られた場合、`PUNCTUATION_DELAY_MS`を1000ms〜1500msに増やすことを検討。

### blockKeyの安定性

Google MeetがDOM要素を再利用するのか、新しい要素を作り直すのかは未確認。
新しい要素が作られると新しい`blockKey`→新しい`segmentId`となり、`finalizedSegments`の追跡が途切れる。
もし同一話者の連続発話で新しいDOM要素が作られるケースがあれば、`blockKey`ではなく話者名ベースの追跡が必要になる可能性がえある。

### `computeTextDelta`の部分文字列検索の精度

`lastText`の末尾句読点を除去して`newText`内を検索する方式のため：

- lastTextが短い（5文字未満）場合はフォールバックが無効（偶然の一致を防ぐため）
- lastTextの内容がnewText内に複数箇所存在する場合、最初のマッチが使われる（`indexOf`）

## 次回の重複問題が起きた場合の調査手順

1. `aaa.txt`のような実データを取得する
2. 重複パターンを確認:

- 全文が毎回出力されている → 仮説3のフォールバックが動いている可能性
- 差分だけが重複している → `computeTextDelta`のマッチが誤っている可能性
- 同じ話者で異なるtimestampが大量にある → finalizeが頻繁に走りすぎ

3. `CaptionEngine`にログを仕込んで、`finalizeSegment`の呼び出し頻度・delta値・フォールバック発生を確認する
4. 特に`computeTextDelta`に渡される`lastText`と`text`の組み合わせを記録し、delta計算が正しく動いているか確認する
