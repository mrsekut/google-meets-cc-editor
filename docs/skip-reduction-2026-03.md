# finalizeスキップ削減の改善記録 (2026-03)

## 背景

dedup-fix-2026-03.md の修正後、重複出力は解消された。
しかし本番ログ（18分間の7人ミーティング）を分析したところ、finalize時の**skip率が23%**（269回中63回）と高く、テキスト欠損が発生していることが判明した。

## ログ分析の主要な発見

| 指標                                       | 値                 |
| ------------------------------------------ | ------------------ |
| 総エントリ数                               | 3,531              |
| セグメント数                               | 55                 |
| finalize回数                               | 269                |
| skip回数                                   | 63 (23.4%)         |
| first回数                                  | 206 (76.6%)        |
| skip-null-delta / revised / skip-identical | 0 (一度も発火せず) |

### skipの主要パターン

**パターン1: 句読点差し替え + テキスト追加（最多）**

音声認識が句読点を一時的に付与し、その後取り消して続きを追加する。

```
確定済: "すごい！"
新テキスト: "すごいリアルで家事しながら。"
→ startsWith失敗、searchable.length < 5 でCase 2も不発 → skip
```

他の例: `"そう！"` → `"そうで、私あの..."`, `"はい。"` → `"はい、えーっと！..."`, `"全然その！"` → `"全然そのライブじゃなかった..."`

**パターン2: 読点の挿入**

```
確定済: "なんかそう書いたりで動物の森と。"
新テキスト: "なんかそう書いたりで、動物の森とブロック系で..."
→ 「で」の後に「、」挿入でstartsWith失敗 → skip
```

**パターン3: 完全書き換え**

```
確定済: "あれ？"
新テキスト: "できないって。" → "ゲーム！"
→ 音声認識が初期認識を完全撤回
```

## 原因

### 原因1: 短いテキストの早期確定

`computeFinalizeDelay`が句読点で終わるテキストに一律500msを返していた。
「そう！」「はい。」「すごい！」のような短い発話は500msで確定されるが、音声認識はその後テキストを書き換える。
書き換え後のテキストは`startsWith`で一致しないためskipになり、テキストが欠損する。

### 原因2: `computeTextDelta`が句読点差し替えパターンに非対応

`"すごい！"` → `"すごいリアルで..."` のように末尾の句読点が消えて続きが追加されるケースで:

- Case 1 (`startsWith`): `"すごいリアル..."` は `"すごい！"` で始まらない → 失敗
- Case 2 (`indexOf`): `searchable = "すごい"` だが `length < 5` → スキップ

句読点を除去した上での前方一致（Case 1.5）が存在しなかった。

## 修正内容

### 修正1: 初回確定 + 短いテキストのディレイ延長

`CaptionEngine.ts` の `computeFinalizeDelay` を改善。
初回確定（`finalizedSegments`にまだ記録がない）かつテキストが10文字未満の場合、句読点があっても2500msで待つ。

これにより「そう！」→ 2500ms待つ間に「そうで、私あの...」に更新され、正しいテキストで初回確定できる。

### 修正2: 句読点除去後の前方一致（Case 1.5）追加

`format.ts` の `computeTextDelta` に新しいケースを追加。

末尾の句読点を除去した`lastText`が`newText`の先頭と一致する場合、その後の部分をdeltaとして返す。

```
lastText: "すごい！" → lastTrimmed: "すごい"
newText: "すごいリアルで家事しながら。"
→ newText.startsWith("すごい") === true
→ delta: "リアルで家事しながら。"
```

Case 2の`searchable`変数もこの`lastTrimmed`を再利用するようリファクタ。

## 期待される効果

ログの58件のskip（テキスト分岐あり）のうち:

- **パターン1（句読点差し替え）**: 修正1で初回確定が遅延され、修正2でdelta計算が成功 → 大半が解消
- **パターン2（読点挿入）**: 修正2で「で」→「で、」のように末尾句読点除去後の前方一致で解消
- **パターン3（完全書き換え）**: 修正1の遅延で書き換え後のテキストで確定される可能性が高い

## 残っている懸念

### 10文字の閾値

`SHORT_TEXT_THRESHOLD = 10` は経験的な値。ログデータでは短いskipの多くが3〜5文字だったため10文字で十分カバーできるが、長めの初回テキスト（8〜9文字）が書き換えられるケースがあれば閾値の調整が必要。

### 完全書き換えへの対処

seg-13のような `"あれ？"` → `"できないって。"` → `"ゲーム！"` の完全書き換えは、修正1の遅延で最終的な認識結果を待てる可能性が高いが、2.5秒以内に安定しない場合は依然としてskipになる。

### Case 2の5文字制限

`searchable.length >= 5` の制限は維持した。3文字に緩和すると誤マッチのリスクがあるため、まず修正1・2の効果を確認してから検討する。
